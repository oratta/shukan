# フィードバック着信の可視化 — 方式検討

## 概要

アプリ内フィードバック（issue #19 / PR #65）は `feedbacks` テーブルへ insert されるだけで、届いたことに誰も気づけません。本文書は「通知（メール / Slack）」と「管理画面での一覧」の各案を比較し、推奨案を 1 つ提示するための設計検討です。実装は本文書の範囲外で、成果物は設計文書のみとします。最終的な採用の決定はレビューで人間が行います。

前提として、通知先の運用者は現状 1 人（個人開発）です。フィードバック本文にはユーザーの個人的な記述が入りうるため、外部サービスへ本文をそのまま流すことの是非を主要な論点に据えます。

---

## 現状

- **RLS は insert-only**。`supabase/migrations/20260706000000_feedbacks.sql` の 14〜22 行目で、`enable row level security` の後に定義されるポリシーは `"Users can insert own feedback"`（`for insert` のみ）1 本だけです。SELECT / UPDATE / DELETE ポリシーは存在しません。
- **読み出しは service_role 限定**。同マイグレーションのコメント（15〜17 行目）に明記されているとおり、SELECT ポリシーが無いため anon・authenticated ロールからは 1 件も読めません。投稿者本人ですら自分の行を読めない設計で、閲覧は RLS をバイパスする service_role のみが可能です。
- **書き込み経路**。`src/lib/supabase/app-feedbacks.ts` の `submitAppFeedback()` がブラウザクライアント（anon key）で `feedbacks` へ insert するのみです。この経路には読み出しも通知もありません。
- **通知基盤は未導入**。`package.json` を確認したところ、`resend` / `nodemailer` / `@slack/*` / webhook 系のいずれの依存も存在しません。メール・Slack を使うには新規に基盤を用意する必要があります。
- **管理画面は無い**。運営が一覧を見る UI は存在せず、現状で本文を見るには Supabase Studio の Table Editor か SQL エディタ（service_role 相当）を手動で開くしかありません。

つまり「届いても気づけない」だけでなく「気づいても見る導線が Studio 手動操作しかない」状態です。本検討は主に前者（着信への気づき）を解決し、後者（閲覧導線）を次アクションで補完します。

---

## 選択肢

以下の 4 案を検討します。案 C・D はさらに「本文を含めて送る」か「メタ情報（カテゴリ・件数・時刻）だけ送り本文は送らない」かの 2 バリエーションを持ちます。

### 案 A: Supabase Studio / SQL で直接閲覧（通知なし）

新規実装をせず、運営が Supabase Studio の Table Editor か SQL で `feedbacks` を service_role 権限で覗く運用です。現状の延長線上にあります。

- 実装: ゼロ。運用手順書を書くだけ。
- 気づき: 運営が能動的に見に行かない限り気づけません。本 issue が解決したい「気づけない」問題がそのまま残ります。

### 案 B: アプリ内管理画面（一覧）

`/admin/feedbacks` のような認証付きページを作り、Server Component または Route Handler で service_role クライアントを使って `feedbacks` を SELECT し、一覧表示します。運営アカウントのみに絞る認可（メールアドレス許可リスト等）が必要です。

- 実装: service_role をサーバー側だけで使う仕組み（環境変数 `SUPABASE_SERVICE_ROLE_KEY`、サーバー専用クライアント）、認可、ページ UI。中規模。
- 気づき: 能動的に開く必要があり、通知は無いため気づきは改善しません。ただし閲覧導線としては Studio より整い、後述の通知案と組み合わせる土台になります。
- 秘匿情報: 本文はアプリ内（自分たちの認証境界内）に留まり、外部サービスへ出ません。

### 案 C: メール通知

Supabase Database Webhook（`feedbacks` の INSERT トリガー）から Edge Function を起動し、`resend` 等のメール API で運営の自分宛にメールを送ります。

- **C-1（本文込み）**: メール本文にフィードバック本文をそのまま載せる。気づきと内容確認が 1 通で完結しますが、ユーザーの個人的記述が外部メール事業者と運営の受信箱を経由します。
- **C-2（メタのみ）**: 「新着フィードバック 1 件（カテゴリ: bug、時刻: …）」だけ送り、本文は載せない。本文は案 B / A で service_role 閲覧する。外部へ本文が出ません。
- 実装: Edge Function 1 本 + Database Webhook 設定 + メール API キー。小〜中規模。
- 気づき: 個人開発者はメールを日常的に見るため、数分〜数時間で気づけます。

### 案 D: Slack 通知

Slack Incoming Webhook（または Bot）へ、案 C と同じく Database Webhook → Edge Function 経由で投稿します。

- **D-1（本文込み）** / **D-2（メタのみ）**: 案 C と同じ 2 バリエーション。本文込みは Slack ワークスペース（外部 SaaS）にユーザーの個人的記述が保存されます。
- 実装: Edge Function + Webhook URL 発行 + Slack 側チャンネル準備。小〜中規模。個人開発者が普段 Slack を常駐利用していない場合、Slack を「そのために開く」運用は続きにくいリスクがあります。
- 気づき: 常駐していれば即時。していなければメールに劣ります。

## 評価軸

4 軸で評価します。凡例: ◎ 良い / ○ 許容 / △ 難あり / × 不可。

| 案 | 実装コスト | 運用コスト | 秘匿情報の扱い | 気づくまでの時間 |
|----|-----------|-----------|----------------|------------------|
| A: Studio 直接閲覧 | ◎ ゼロ | △ 毎回手動で見に行く | ◎ 外部へ一切出ない（service_role 手動閲覧） | × 能動的に見ない限り気づけない |
| B: 管理画面 | △ service_role 基盤 + 認可 + UI | ○ 開けば一覧できる／通知は無い | ◎ 認証境界内に留まる | × 通知が無く気づきは改善しない |
| C-1: メール（本文込み） | ○ Edge Function + メール API | ◎ 受信箱を見るだけ | × 本文が外部メール事業者・受信箱を経由 | ◎ 数分〜数時間 |
| C-2: メール（メタのみ） | ○ Edge Function + メール API | ○ 通知で気づき→本文は別途閲覧 | ○ 本文は外部へ出ず、メタのみ送出 | ◎ 数分〜数時間 |
| D-1: Slack（本文込み） | ○ Edge Function + Webhook | ○〜△ 常駐なら楽／非常駐なら見落とす | × 本文が外部 SaaS に保存される | ○ 常駐時は即時／非常駐は劣る |
| D-2: Slack（メタのみ） | ○ Edge Function + Webhook | ○〜△ 同上 | ○ 本文は外部へ出ず、メタのみ送出 | ○ 常駐時は即時／非常駐は劣る |

補足:

- **秘匿情報**が本検討の要です。本文にはユーザーの心情・体調・固有名詞などが入りうるため、本文を外部 SaaS（メール事業者・Slack）へ流す C-1 / D-1 は、たとえ運営 1 人しか見なくても「第三者事業者のサーバーに個人的記述が保存される」点で望ましくありません。メタのみ送出（C-2 / D-2）なら着信の気づきを得つつ本文を外部へ出さずに済みます。
- **気づくまでの時間**を根本改善できるのは通知系（C / D）だけです。A / B は閲覧導線であり、能動的に見に行く必要があるため「気づけない」という本 issue の課題は解けません。
- **個人開発 1 人**という前提では、常駐前提の Slack より、既に日常的に見ているメールのほうが見落としが少なく運用が続きます。

## 推奨

**採用案: 案 C-2 — メタ情報のみのメール通知**（本文は送らず、カテゴリ・件数・時刻・投稿 ID だけを運営の自分宛にメールする）。

### 理由

1. **本 issue の核心「気づけない」を解く**。通知系のみが気づくまでの時間を根本改善でき、その中でメールは個人開発 1 人が最も確実に日常閲覧するチャネルです。Slack 常駐を前提にできない個人開発では、メールのほうが見落としが少なく運用が長続きします。
2. **秘匿情報を外部へ出さない**。メール本文にフィードバック本文を載せないため、ユーザーの個人的記述が外部メール事業者や受信箱に残りません。メールで運ぶのは「新着があった」という事実とメタ情報だけです。
3. **実装が軽い**。Supabase Database Webhook → Edge Function → メール API という、既存の Supabase 基盤に閉じた小さな追加で済みます。新たな常駐サービスを増やしません。

### 本文の閲覧について

本文は通知に載せず、当面は現状どおり Supabase Studio（service_role）で確認します。通知で「見るべき件がある」と分かってから Studio を開くため、A 単独運用の「そもそも気づけない」問題は解消されます。恒久的な閲覧導線としては案 B（管理画面）を次アクションの後続 issue として切り出し、通知（気づき）と管理画面（閲覧）を段階的に揃えます。

### 採らなかった案とその理由

- **案 A（Studio 直接閲覧のみ）**: 通知が無く「気づけない」がそのまま残るため、本 issue の解決になりません。ただし本文閲覧の暫定手段として推奨案に内包します。
- **案 B（管理画面のみ）**: 閲覧導線は整いますが通知が無く、能動的に見に行かないと気づけません。単独では課題を解けないため、推奨案の後続（閲覧導線の恒久化）に位置づけます。
- **案 C-1 / D-1（本文込み通知）**: 気づきは最速級ですが、ユーザーの個人的記述を外部 SaaS に流す点が前提（秘匿情報の尊重）と相反するため不採用です。
- **案 D-2（メタのみ Slack）**: 秘匿性は C-2 と同等ですが、個人開発で Slack 常駐を前提にできず、非常駐時に見落とすリスクがメールに劣ります。将来チーム運用に移行した際の再検討候補とします。

## 次アクション

推奨案（C-2）と、その閲覧導線（案 B）を後続 issue のドラフトとして起こします。

### 後続 issue ドラフト 1（主）: メタ情報のみのメール通知を実装する

**タイトル**: `feat: フィードバック着信をメール（本文非送出・メタのみ）で通知する`

**受け入れ条件の骨子**:
- `feedbacks` への INSERT を Supabase Database Webhook で捕捉し、Edge Function を起動する
- Edge Function は運営の自分宛にメールを送る。**メール本文にフィードバック本文（`body` 列）を含めない**。含めるのはカテゴリ（`category`）・投稿時刻（`created_at`）・投稿 ID（`id`）・件数などのメタ情報に限る
- メール API キー等の秘匿値は環境変数・Supabase secrets で管理し、リポジトリにコミットしない
- 送信失敗時にフィードバックの insert 自体は成功させる（通知失敗が投稿をブロックしない）
- テスト insert で実際にメールが届くことを 1 度確認する（手順を PR に記載）

### 後続 issue ドラフト 2（従）: フィードバック閲覧用の管理画面を作る

**タイトル**: `feat: 運営用フィードバック一覧ページ（/admin/feedbacks）を追加する`

**受け入れ条件の骨子**:
- サーバー側（Server Component または Route Handler）でのみ service_role クライアントを使い、`feedbacks` を新しい順に SELECT して一覧表示する。service_role キーをクライアントバンドルに絶対に含めない
- 運営アカウント（許可リストのメールアドレス等）以外はアクセスできない認可をかける
- カテゴリでの絞り込みと本文表示ができる
- 本ページ追加に伴い、`feedbacks` の RLS ポリシー（insert-only）は変更しない（読み出しは引き続き service_role 経由に限定する）
