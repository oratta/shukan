## Context

Shukanアプリは習慣トラッカーとして基本的なCRUD + 完了トグル + ストリーク追跡を実装済み。
既存のインパクト表示機能（1習慣=1記事）を拡張し、マルチエビデンス対応のマーケットプレイスを構築する。

技術スタック: Next.js 16 + React 19 + Supabase + Tailwind CSS 4 + Shadcn UI

## Goals / Non-Goals

**Goals:**
- エビデンスベースの記事で習慣継続のモチベーション向上
- 1習慣に複数エビデンスを紐付ける柔軟なアーキテクチャ
- Discoverタブから1タップで習慣作成（フリクション最小化）
- 既存データとの完全な後方互換性

**Non-Goals:**
- LLM推論によるパーソナライズ（V3以降）
- ユーザー生成エビデンスの作成UI
- エビデンスのお気に入り・ブックマーク機能
- エビデンス記事の高度な検索・フィルタリング

## Decisions

### Decision 1: ジャンクションテーブル方式
- **選択**: habit_evidencesテーブルでN:M関係を管理
- **理由**: 重み（weight）を各紐付けに持たせる必要があり、JSON配列やArrayカラムでは不十分
- **トレードオフ**: 追加JOIN必要だが、Supabaseのrelationで`select('*, habit_evidences(*)')`で効率的に取得可能

### Decision 2: weight は整数パーセンテージ（1-100）
- **選択**: 浮動小数点ではなく整数パーセンテージ
- **理由**: UIスライダーで5刻み調整、精度は十分、表示が直感的
- **トレードオフ**: 0.5%刻みは不可だが、実用上問題なし

### Decision 3: 後方互換性の維持
- **選択**: impact_article_idカラムを残し、evidences優先 → fallback
- **理由**: 段階的移行。既存ユーザーデータを壊さない
- **トレードオフ**: 冗長なデータだが、安全性を優先

### Decision 4: エビデンス記事はフロントエンドに静的保持
- **選択**: TypeScriptファイルとして `src/data/impact-articles/` に配置
- **理由**: 記事数が少ない（7件）、頻繁な更新なし、ビルド時型安全性
- **トレードオフ**: 記事追加にはデプロイが必要

### Decision 5: Pencil MCPでUIデザイン先行
- **選択**: Phase 2でPencil MCPによるモックアップ作成・レビュー後に実装
- **理由**: ユーザーフィードバックを早期に取得し、手戻りを防止
- **結果**: 11画面のデザインを作成し、2e までの5回のイテレーションを実施

## Risks / Trade-offs

### リスク1: エビデンス記事の信頼性
- 科学的出典を引用しているが、査読済み論文のみではない
- **対策**: confidenceLevel（high/medium/low）で透明性を確保

### リスク2: weight合算が100%を超える
- 2つのエビデンスが各weight=100の場合、200%相当の重複計算になる
- **対策**: 仕様上許容。各エビデンスの「適用率」という位置づけで、合計100%の制約は不要
