# Design: 計算ロジック表示 + フィードバック機能

## Context

エビデンス記事の数値の透明性を高め、ユーザーからの疑問報告を収集する機能。禁煙記事の dailyHealthMinutes 計算ミス（12→288）の発見を契機に、全30記事の整合性検証も同時に行う。

## Goals

- 計算根拠を構造化データで保持し、UIで透明に表示する
- ユーザーからのフィードバック（バッドマーク + コメント）を Supabase に保存する
- 全30記事の calculationParams の数学的整合性を保証する

## Non-Goals

- フィードバック集計結果のユーザー向け表示（将来のLLM調査用のみ）
- ユーザー自身による calculationParams の編集
- V3 動的プロフィール対応（V2 ハードコードプロフィール固定）

## Decisions

### D1: 計算ロジックのデータ形式
**決定**: 構造化データ（CalcStep 配列）
**理由**: プレーンテキストよりも検証可能で、将来的な自動チェックにも対応できる。UIでの表示制御も容易。

### D2: 健康寿命の計算方式
**決定**: `効果年数 × 525,600分 ÷ 残存寿命年数 ÷ 365日` で日割り
**理由**: 統計的期待値をそのまま日割りする方式。数値が大きくなるが、期待値であることを明記して透明性を確保する。

### D3: フィードバック保存先
**決定**: Supabase テーブル + 集計View
**理由**: クロスデバイス対応と将来のLLM調査用データ蓄積のため。

### D4: バッドマークの仕様
**決定**: トグル式（1ユーザー1記事1回まで）
**理由**: スパム防止。UNIQUE制約ではなく、既存レコードの INSERT/DELETE で実装。

### D5: article_feedback_stats View のアクセス
**決定**: service_role キーでのみアクセス（将来のLLM調査用）
**理由**: RLSが有効なため、anon キーでは自分のデータしか見えない。全体集計は管理用途のみ。クライアントサイドの feedbacks.ts からはこの View を呼び出さない。

### D6: calculationLogic の optional 設計
**決定**: calculationLogic は optional フィールド（`calculationLogic?:`）
**理由**: 型定義追加（Task 1）→ UI実装（Task 3）→ 記事データ追加（Task 4）の順序で進めるため、全記事にデータが揃う前にビルドが通る必要がある。UIは `article.calculationLogic &&` で guard する。

### D7: i18n キー構造
**決定**: 既存の `"evidence"` 名前空間にフラットに追加
**理由**: 既存パターン（`evidence.addThisHabit` 等）と一貫性を保つ。feedback 関連キーは `evidence.feedback*` プレフィックスで区別する。

## Risks

- **R1**: dailyHealthMinutes の修正で数値が大幅に変わる記事がある（例: 12→288）。ユーザーの既存の印象と乖離する可能性。
  - 対策: calculationLogic で計算過程を透明に表示し、期待値であることを明記
- **R2**: 30記事分の計算ロジック作成は作業量が大きい
  - 対策: 統一ルールに基づき機械的に処理。矛盾検出は自動化
