# Design: 頻度サポート（Daily / Weekly 対応）

## Context

現在すべての習慣は `frequency: 'daily'` で固定されている。型定義に `'weekly'` と `'custom'` は存在するが、`shouldShowToday` での `weekly` 処理は月曜固定の未完成実装であり、実際には機能していない。HabitForm の頻度セレクターも非表示状態にある。

ユーザーが「平日だけ」「週3回」などの現実的な習慣頻度を設定できるようにすることが本変更の目的。

## Goals

- 平日限定（weekday）・特定曜日（custom）・週N回（weekly）の習慣頻度を完全サポートする
- 非対象日の習慣を自動的にスキップセクションに配置し、UX の混乱を防ぐ
- 既存の everyday 習慣に一切の影響を与えない
- 既存のスキップ機構（habit-skip）を最大限再利用し、新しいメカニズムを作らない

## Non-Goals

- カレンダービュー（HabitDetailModal）の頻度対応（非対象日はスキップ日と同じ扱いで既存処理に委ねる）
- Impact 計算の頻度対応（completedDays ベースなので自動的に正しい、変更不要）
- 月次・年次などの週未満・週超えの頻度サポート

## Decisions

### D1: 新カテゴリ構造（everyday / weekday / custom / weekly）

**決定**: 従来の `daily` を廃止し、4種別に再編する
- `everyday`: 毎日（旧 `daily` 相当、デフォルト）
- `weekday`: 平日限定（月〜金）
- `custom`: 特定曜日指定（`customDays: number[]` で管理）
- `weekly`: 週N回（曜日問わず、`weeklyTarget` で目標回数管理）

**理由**: 旧 `weekly`（月曜固定）と旧 `custom` の設計意図が実装と乖離していた。新構造では Daily 系（毎日系）と Weekly 系（週N回）を明確に分離し、将来の拡張も容易にする。

### D2: 非対象日の処理は既存スキップ機構を再利用

**決定**: `shouldShowToday` を非アーカイブなら常に `true` に簡素化し、非対象日の習慣には `skippedToday: true` を自動付与する

**理由**: 既存の habit-skip 機構（DB に `skipped` レコードを書き込み、HabitList でセクション分割表示）はすでに動作している。新しい非表示メカニズムを作るよりも、この機構を再利用することで実装量を最小化し、UX の一貫性を保てる。

**重要**: 自動スキップは DB にレコードを書き込まない。表示ロジック上のみで判定する。ユーザーが手動で unskip すると `setDayStatus(id, today, 'none')` が呼ばれ明示的な `none` レコードが作られるため、以後は自動スキップが上書きされない。

### D3: Weekly 習慣はホーム画面に毎日表示

**決定**: `weekly` 習慣も `shouldShowToday` で毎日 `true` を返し、毎日ホーム画面に表示する

**理由**: Weekly 習慣は「週のどの日にやってもOK」というセマンティクスを持つ。毎日表示することでユーザーが好きなタイミングで記録できる。目標達成後の表示はカード上の進捗（今週 N/M 回）で判断できる。

### D4: 週の区切りは月曜始まり（ISO 8601）

**決定**: Weekly のストリーク計算・completionRate 計算の週区切りを月曜始まりとする

**理由**: 国際標準（ISO 8601）に準拠する。多くのビジネス・フィットネスアプリも月曜始まりを採用しており、ユーザーの直感と一致する。

### D5: 自動スキップの優先順位（ユーザー操作 > 自動判定）

**決定**: 自動スキップの判定は「今日のレコードが存在しない かつ isTargetDay = false」の場合のみ適用する。ユーザーが手動で完了・失敗・unskip した場合は自動スキップを上書きしない。

**理由**: ユーザーが意図的に非対象日にも習慣を実施したいケースに対応するため。自動スキップはあくまでデフォルト挙動であり、ユーザー操作が最優先。

```
skippedToday の判定優先順位:
  1. 今日のレコードが status='skipped' → true（ユーザー手動スキップ）
  2. 今日のレコードが status='completed'/'failed'/'rocket_used'/'none' → false（ユーザー操作済み）
  3. 今日のレコードがない + isTargetDay(habit, today) = false → true（自動スキップ）
  4. 今日のレコードがない + isTargetDay(habit, today) = true → false（通常の未実施）
```

### D6: Weekly ストリーク計算は連続達成週数

**決定**: Weekly 習慣のストリークは日単位ではなく週単位で計算する。1週間の中で `weeklyTarget` 回以上 completed があれば「達成週」とみなす。

**理由**: Weekly 習慣に日単位のストリークは意味をなさない（週3回なら毎日実施することを期待しない）。連続達成週数の方がユーザーの努力を正確に反映する。

### D7: Weekly completionRate は過去12週ベース

**決定**: Weekly 習慣の completionRate = 過去12週のうち達成週の数 / 12

**理由**: Daily 系の30日ウィンドウに相当する期間として12週（約3ヶ月）を採用。週次習慣の傾向把握に適切な粒度。

### D8: HabitForm は `frontend-design` スキルを使用

**決定**: 頻度セレクター UI の実装は実装エージェントが `frontend-design` スキルを使用する

**理由**: カテゴリタブ・サブタイプラジオ・曜日チップ・回数セレクターと複雑な UI コンポーネントが必要なため、デザイン品質を担保する専用スキルを使用する。

## Risks

- **R1**: 既存 `everyday` 習慣（旧 `daily`）への影響
  - 対策: DB マイグレーションで `'daily'` → `'everyday'` に一括変換。`isTargetDay` の `everyday` ケースは無条件 `true` を返すため既存動作と完全互換
- **R2**: 自動スキップと手動スキップの競合
  - 対策: D5 の優先順位ロジックで明確に定義。今日のレコードが存在する場合は自動スキップを適用しない
- **R3**: Weekly ストリーク計算の今週進行中ケース
  - 対策: 今週が進行中でも `weeklyTarget` 達成済みならカウントに含める。未達成なら今週はまだカウントしない（連続は途切れていない状態を維持）
