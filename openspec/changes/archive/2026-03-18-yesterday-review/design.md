## Context

ホーム画面には DailyImpactSummary と習慣リストが表示されている。ユーザーは翌日以降に昨日の習慣状態をまとめて確認・修正したいが、現状は個別ドットを探す必要がある。昨日分の一括レビュー機能を最小限の UI 変更で追加する。

## Goals

- 昨日の未レビュー習慣がある場合にのみ目立つ通知を表示
- ホーム画面から離れずに昨日の習慣を一括レビューできる
- ムードと日次コメントで振り返りの質を高める
- 5日超過の未記録は自動的に failed 扱いにして集計の一貫性を保つ

## Non-Goals

- 2日以上前の日付のレビュー UI（昨日のみ対象）
- プッシュ通知によるレビュー促進
- レビュー完了後の達成バッジ・アニメーション
- ムードの履歴グラフ表示

## Decisions

### D1: バナー + Sheet パターン（新規ルート/タブを作らない）
**選択**: ホーム画面に Amber バナーを追加し、タップで Sheet を開くパターン。
**理由**: ホームを唯一のエントリーポイントとして維持することで、ナビゲーション構造を複雑化しない。Sheet は既存の Shadcn UI コンポーネントを流用でき、実装コストが低い。

### D2: 頻度非依存のレビュー対象
**選択**: 非アーカイブの全習慣を昨日のレビュー対象とする（weekly/custom の頻度は考慮しない）。
**理由**: 「昨日この習慣をやったか」という問いに頻度制約は不要。ユーザーが自分で skipped を選択できるため、対象外日の習慣も表示して構わない。頻度フィルタリングはロジックを複雑化するだけでメリットが薄い。

### D3: status の即座保存（確認ダイアログなし）
**選択**: status トグルタップ時に `setDayStatus()` を即時呼び出して DB に保存する。
**理由**: 習慣の status 変更は既にホーム画面のドットで即時保存されており、同じ UX を維持することでユーザーの学習コストを下げる。誤タップは再タップで訂正できる。

### D4: メモのblur-to-save パターン
**選択**: per-habit メモは input の onBlur イベントで `updateCompletionNote()` を呼び出して保存する。
**理由**: キーストロークごとの保存は DB 負荷が高い。「Done」ボタンまで待つと閉じ忘れ時にデータが消える。blur-to-save が両者のバランスを取れる。

### D5: ムード + コメントはローカル状態を保持し「Done」で保存
**選択**: ムードスタンプとコメントは Sheet 内でローカル state に保持し、「Done」ボタン押下時に `upsertDailyReflection()` で保存してシートを閉じる。
**理由**: ムードとコメントは1日単位の確定情報であり、記入中の中間状態を逐次保存する必要がない。「Done」で確定する UX がより自然。

### D6: none の意味論 — DB は none を保持し、読み取り層で変換
**選択**: DB には `none` のまま保存する。`getEffectiveStatus()` 関数が5日超過の `none` を `failed` に変換し、集計時に使用する。
**理由**: DB の生データを書き換えると、ユーザーが後から「実は記録してなかっただけ」という状態に戻せなくなる。読み取り時のみ変換することで、実際の記録状態と集計上の扱いを分離する。

### D7: daily_reflections テーブルを habit_completions とは別テーブルに設ける
**選択**: ムード + コメントを `daily_reflections` テーブル（user_id + date の複合ユニーク）で管理する。
**理由**: ムード・コメントは習慣単位ではなく日次単位でユーザーに紐付く情報。habit_completions に混在させると 1:N の関係が崩れる。

## Risks

- Sheet 内でステータスを変更すると、ホーム画面の習慣リスト表示がリアルタイム更新されない可能性がある → Sheet クローズ時に habits を再フェッチするか、楽観的更新を採用して対処する。
- `getYesterdayUnreviewedHabits()` のクエリが習慣数に比例して重くなる可能性 → 現状の習慣数規模では問題なし。将来的にはバッチ集計を検討。
