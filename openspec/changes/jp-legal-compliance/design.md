# Design: jp-legal-compliance

## Context

Smitch は日本（ja）と英語圏（en）の双方に提供する Web アプリで、change-A で Stripe 課金（月額 $4.99 / 年額 $39.99 / Lifetime $99 既定）、change-B で Founding Member 割引、change-C でティザー + waitlist が入る。日本の消費者に課金機能を提供する以上、改正特定商取引法（2022 年施行の最終確認画面規制 第12条の6）・消費税法の総額表示義務・景品表示法・個人情報保護法への準拠は公開前の必須要件であり、Stripe は自動対応しないため自前実装する。

既存資産:
- `/privacy`（`src/app/privacy/page.tsx`）: 全12セクションの日本語プライバシーポリシー。収集情報（§2）・利用目的（§3）・データ保存（§4）・問い合わせ（§12）あり。決済・waitlist への言及なし
- `/terms`（`src/app/terms/page.tsx`）: 全12セクションの日本語利用規約。§2 に「本サービスは現在、無償で提供されています」、§8 に「本サービスは無償で提供されており」という記述があり、課金開始後は実態と矛盾する
- 両ページとも Server Component + Tailwind セマンティッククラス + `<section>` 構造で統一されている

## Goals / Non-Goals

**Goals:**
- 特商法表記ページの新設と、価格表示面からの到達導線
- Checkout 直前の最終確認画面に必須4項目（定期購入である旨／各回代金+支払総額／トライアル移行時期と金額／解約方法・期限・違約金）をスクロール不要で表示し、テストで存在検証
- すべての消費者向け価格表示を税込総額に統一（表示額 = 請求額）
- Customer Portal 経由の解約導線と「いつでも解約」表示の一致
- 残り枠・割引表示が change-B の実数に基づくことのテスト担保
- 既存 `/privacy` `/terms` への課金・waitlist 条項の追記（新規ページは作らない）

**Non-Goals:**
- 海外（EU VAT 等）の税務対応・MoR 切替（plan.md の将来判断）
- 資金決済法対応（前払式支払手段は扱わない。Lifetime は役務提供の一括対価として整理）
- 弁護士レビューの代替（実装はガイドライン準拠だが、最終的な法的確認は人間が行う）
- 解約時アンケート・リテンションオファー等の解約フロー拡張

## Decisions

### D1: 特商法表記ページは `src/app/tokushoho/page.tsx`（トップレベル既存パターン踏襲）
- **判断**: `/privacy` `/terms` と同じトップレベル route + Server Component + 同一のレイアウト/スタイル構造で `/tokushoho` を新設。`<dl>` または section 構造で必須項目を列挙
- **代替案 A**: `/legal/tokushoho` のような `legal/` 配下へ集約（privacy/terms も移動）
- **却下理由**: 既存ページの URL 変更はリンク切れ（LP footer・OAuth 同意画面の登録 URL 等）のリスクがあり、本 change の責務外。既存のフラット構造に合わせるのが最小
- **補足**: 事業者名・所在地・連絡先の実値（Genetta Inc. の登記情報）は実装時にユーザーへ確認する。プレースホルダのままリリースしない

### D2: 最終確認画面は自前ページ（Checkout リダイレクト前の中間ステップ）として実装
- **判断**: 「プラン選択 → 最終確認画面（自前、必須4項目表示）→ 確認ボタン → `/api/stripe/*` で Checkout セッション生成 → Stripe へリダイレクト」のフローとし、最終確認画面を change-A の Checkout フロー手前に挿入する。画面はテスト容易性のため、必須4項目を props（プラン・価格・トライアル状態）から純粋に描画するコンポーネントに分離し、Vitest の構造テスト（lp-shell-and-copy で確立した tree-walking 方式）で4項目の存在を assert する
- **代替案 A**: Stripe Checkout 画面自体（custom_text 等）に法定表示を寄せる
- **却下理由**: Stripe Checkout のカスタマイズでは「申込み最終確認画面に一目で見える」要件の充足を自分で保証できず、表示文言・配置の管理が Stripe 側仕様に従属する。改正特商法のスクリーンショット保存実務でも自前画面が安全
- **補足**: 必須4項目はボタン直上に固定配置し、アコーディオン・モーダル・タブの背後に置かない（spec のスクロール不要要件）

### D3: 税込表示は「Stripe Price を税込（tax_behavior: inclusive）で設定」し、Stripe Tax の自動計算は導入しない
- **判断**: 価格既定値（月額 $4.99 / 年額 $39.99 / Lifetime $99）を税込総額として扱い、Stripe Price の `tax_behavior` を `inclusive` に設定。UI 表示は「$4.99（税込）/ 月」のように税込であることを明示し、表示額 = 請求額を常に一致させる。通貨は USD のまま（Stripe Checkout がローカライズ表示を担う）
- **代替案 A**: Stripe Tax を導入し税抜価格 + 自動税計算（exclusive）
- **却下理由**: 総額表示義務は「表示時点で税込総額が分かること」が本質であり、exclusive にすると表示面ごとに税込換算ロジックが必要になり表示額と請求額の乖離リスクが増える。Stripe Tax は登録・申告体制が必要で現フェーズでは過剰。将来 MoR/Stripe Tax へ移行する場合も Price 定義の差し替えで対応できる
- **代替案 B**: JPY 建ての別 Price を用意する多通貨対応
- **却下理由**: 価格管理が二重になり Founding 割引（change-B）との組合せが複雑化する。まず単一通貨で検証する plan.md の方針に従う

### D4: 解約導線は Customer Portal 一本化 + 表示文言の一致をテスト
- **判断**: 解約は change-A の Customer Portal 連携をそのまま使い、アカウント/課金画面に Portal リンクを常設。「いつでも解約できます」という文言を使う場合は、追加条件（電話必須・期限縛り等）を一切付けない。Portal 設定でサブスクキャンセルを有効化することを change-A 側のセットアップ要件として明記
- **代替案 A**: 自前の解約フロー実装
- **却下理由**: Portal で要件（簡単に解約できる・表示と実態の一致）は満たせる。自前実装は工数とバグリスクだけ増える

### D5: 景表法検証は「表示が実数ソースに接続されていること」のテストとして実装
- **判断**: 残り枠表示コンポーネントが change-B のカウンタ API（公開エンドポイント）から値を受け取る構造であること、割引率表示が実販売価格（通常 Price）を参照価格とすることを、ユニットテストで検証する（ハードコード数値の検出）。偽カウントダウン・架空在庫はコードレビュー + config.yaml ルールでも禁止
- **代替案 A**: E2E で表示値と DB 値の一致を毎回検証
- **却下理由**: E2E は change-B 統合後の確認には有効だが、本 change 単体ではモック境界（カウンタ API の返り値 → 表示）の構造テストで担保するのが安定。統合検証は受け入れ条件 20 の統合フェーズで行う

### D6: privacy/terms は既存ページへの「追記」方式（構造維持）
- **判断**: `/privacy` は既存12セクション構造を維持したまま、§2（収集する情報）に waitlist メールアドレスと Stripe へ送信される決済情報を追加、§4 または新設セクションで「決済処理は Stripe, Inc. に委託しカード情報は当方サーバーに保存しない」旨と Stripe プライバシーポリシーへのリンクを追記、§3 に課金管理・waitlist 案内送付の利用目的を追加する。`/terms` は §2 と §8 の「無償で提供」記述を有料プラン併存の実態に書き換え、料金・解約の詳細は特商法表記ページへ誘導する
- **代替案 A**: ポリシーページの全面リライト
- **却下理由**: plan.md が「新規作成ではなく既存ページに追記」と明示。既存文言は OAuth 審査等で参照されている可能性があり、差分を最小にして最終更新日を更新するのが安全

### D7: 法定表示ページは日本語を正とする（en 翻訳は参考扱い）
- **判断**: 特商法表記・最終確認画面の法定文言は日本語で実装する。最終確認画面は next-intl の枠組みに乗せるが、ja 文言を法的要件の正とし、en は同内容の参考訳とする
- **代替案 A**: en のみ／ja のみ
- **却下理由**: 法令の名宛人は日本の消費者向け表示なので ja は必須。一方アプリは en/ja 両対応（next-intl）なので、確認画面だけ ja 固定にすると UX が割れる

## Risks / Trade-offs

| Risk | Mitigation |
|------|------------|
| 事業者の所在地・連絡先など実値が未確定のまま実装が進む | tasks に「ユーザーへの実値確認」を独立タスク化。プレースホルダ検出テスト（`TODO`/`XXX` が tokushoho ページに残っていたら fail）を入れる |
| change-A の Checkout フロー未完成だと最終確認画面が結合できない | 確認画面コンポーネントは props 駆動の純粋コンポーネントとして先行実装・先行テストし、結合は change-A 完了後のタスクに分離 |
| USD 建て「税込」の整理が消費税実務と乖離する懸念 | 表示額 = 請求額（inclusive）を守ることが総額表示の趣旨。税務申告の整理は Non-Goal とし、将来 Stripe Tax / MoR 導入時に再設計する余地を D3 に明記 |
| 「いつでも解約」と Portal 実態の乖離（Portal 設定漏れ） | change-A のセットアップ手順に Portal のキャンセル有効化を明記し、動作確認手順（テストモードで解約完走）を tasks に含める |
| 残り枠の実数検証が change-B 未完成でブロックされる | 本 change では「表示がカウンタ API に接続される構造」のユニットテストまでとし、実数との突合は統合フェーズへ送る（D5） |
| 法解釈の誤り（ガイドライン読み違え） | 消費者庁ガイドライン・Stripe の特商法サポートドキュメントを research-with-fallback で参照して文言を作成し、最終的な弁護士確認は人間タスクとして残す |

## Migration Plan

1. **独立着手分（依存なし）**: 特商法表記ページ新設 → privacy/terms 追記 → 各構造テスト。この時点でデプロイしても既存機能への影響はない（静的ページ追加・文言更新のみ）
2. **change-A 結合分**: 最終確認画面コンポーネントを Checkout フロー手前に挿入。Stripe Price の `tax_behavior: inclusive` 設定は change-A の Price 作成時に適用
3. **change-B 結合分**: 残り枠・割引表示の実数接続テストを有効化
4. **ロールバック**: 表記ページ・ポリシー追記は git revert で戻せる。最終確認画面を外す場合は Checkout 直行に戻るが、その状態では日本向け課金を公開しないこと（法令未準拠状態でのリリース禁止）

## Open Questions

- 事業者表記の実値（屋号/法人名・所在地・電話番号の省略可否＝請求があれば遅滞なく開示する方式を使うか）→ 実装時にユーザー確認
- 年額プランの「一定期間の支払総額」の表現粒度（年額一括なので各回代金 = 年間総額。月額は「月額 $X.XX、年間合計 $YY.YY 相当」まで書くか）→ 文言ドラフト時に確定
